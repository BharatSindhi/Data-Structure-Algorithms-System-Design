DSA Interview Approach Guide — step-by-step framework, patterns, and interview tips

Follow this exact order every time — whether practicing or in a live interview.
1

Understand the problem

Read slowly. Identify: input type, output type, constraints. Don't rush to code. Ask yourself — what exactly is being asked?

2-3 min
2

Work through examples

Take the given examples and trace them manually. Then create your own edge cases: empty input, single element, all same values, negative numbers, duplicates.

3-5 min
3

Brute force first

State the naive solution out loud. Don't code it — just say "I can do this in O(n²) with nested loops." This proves you understand the problem and gives a baseline to improve.

1-2 min
4

Identify the pattern

Ask: Is this a sliding window? Two pointers? BFS/DFS? DP? This is the key step. Match constraints to patterns — see the Pattern tab for a quick guide.

2-3 min
5

Plan the algorithm (pseudocode)

Write steps in plain English or pseudocode before touching actual code. Say it out loud in interviews. This catches logical errors early and shows structured thinking.

3-4 min
6

Code it up

Write clean code. Use meaningful variable names. Handle edge cases as you go. Don't optimize prematurely — get it working correctly first.

10-15 min
7

Test & optimize

Run through your examples manually on the code. Check edge cases. State the time and space complexity. Then discuss potential optimizations if time allows.

3-5 min

Constraint → Expected complexity

  • n ≤ 20 → O(2ⁿ) or O(n!) ok (backtracking)
  • n ≤ 500 → O(n²) ok
  • n ≤ 10⁴ → O(n log n) or O(n) needed
  • n ≤ 10⁶ → O(n) or O(log n) required
  • n ≤ 10⁹ → O(log n) or O(1) only
Think out loud. Silence = fail. Say everything.

Before you code

  • Repeat the problem back in your own words
  • Clarify ambiguities — don't assume
  • Ask about constraints (size, range, nulls)
  • State brute force, then say "I'll optimize"

While coding

  • Narrate every step you write
  • Name variables clearly, not a, b
  • Write helper functions if logic is complex
  • Don't panic — pause, think, then speak
Stuck? Don't freeze. Say "Let me think about this differently…"

After you code

  • Trace through example on paper/whiteboard
  • Check edge cases explicitly
  • State time + space complexity
  • Suggest follow-up improvements

When stuck — rescue moves

  • Draw it out (tree, graph, array)
  • Solve smaller version first
  • Ask "what info do I have at step i?"
  • Try every DS: stack? queue? map?
